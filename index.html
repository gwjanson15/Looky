Rebalance Schedule

Quarterly rebalancing aligned with 13F filing periods:
Q1: Rebalance on ~April 1 (after March 31 period end)
Q2: Rebalance on ~July 1 (after June 30 period end)
Q3: Rebalance on ~October 1 (after September 30 period end)
Q4: Rebalance on ~January 2 (after December 31 period end)

Note: 13F filings are due 45 days after quarter end. This strategy rebalances at quarter boundaries, then updates weights when the new 13F becomes available.

Alpaca Integration

Connect to Alpaca for paper or live trading. Set these environment variables:

ALPACA_API_KEY=your_key_here ALPACA_SECRET_KEY=your_secret_here ALPACA_BASE_URL=https://paper-api.alpaca.markets DEPLOY_CAPITAL=100000

Paper trading recommended for testing

Railway Deployment

One-click deploy to Railway:

1. Push to GitHub 2. Connect repo in Railway dashboard 3. Set environment variables 4. Railway auto-detects Procfile 5. Deploy!

Architecture

  • Flask API backend (app.py)
  • Alpaca trading module (alpaca_trader.py)
  • Quarterly scheduler (scheduler.py)
  • Static HTML/JS dashboard
  • Docker + Procfile for Railway

API Endpoints

  • GET /api/holdings
  • GET /api/allocate?capital=N
  • GET /api/backtest?capital=N&years=5
  • GET /api/rebalance-preview
  • GET /api/alpaca-config

Safety Guardrails

  • Always paper-trade first
  • Max 15% single-position limit
  • Dry-run mode by default
  • Rebalance confirmation required
  • Duplicate rebalance protection
